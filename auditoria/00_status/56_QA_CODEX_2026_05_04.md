## 6. Eixo 5 — Onda 18 crítica

### 6.1 Validação da Opção B do operador

**Concordo com a Opção B.** Separar `ContarStrikesPorEmpresa` como histórico total e criar `ContarStrikesParaPunicao` como janela vigente é a solução correta para preservar auditoria e evitar dupla penalização efetiva.

Pontos fortes:

- Mantém transparência regulatória: histórico completo continua consultável.
- Reduz blast radius: o fluxo de punição muda em um ponto de `Svc_Avaliacao`, sem apagar dados.
- Cria semântica explícita: "memória histórica" não é o mesmo que "punição vigente".

Divergências/ajustes necessários:

- A spec precisa dizer qual data de `CAD_OS` define "pós-reativação". Hoje não há campo separado `DT_AVALIACAO`; o campo persistido usado pela avaliação é `COL_OS_DT_FECHAMENTO` em `Repo_Avaliacao.Inserir` (`src/vba/Repo_Avaliacao.bas:66-70`). Se o operador informa uma data passada, isso muda o corte temporal. A decisão deve ser explícita.
- A spec deve cobrir `Reativa_Empresa.frm`. Ela não chama `Svc_Rodizio.Reativar`; copia/deleta linhas diretamente. Se `DT_ULT_REATIV` entrar só no serviço, a UI real pode não alimentar a janela.
- A migração "campo vazio = legado" é compatível, mas não recupera empresas já reativadas antes da migração. Se houver eventos `EVT_REATIVACAO` no `AUDIT_LOG`, vale backfill controlado da última reativação.
- `GravarStatusEmpresa` precisa deixar de ser Sub silenciosa ou a Onda 18 pode gravar `DT_ULT_REATIV` sem saber se persistiu.

### 6.2 Proposta técnica detalhada para MD-18.1

1. **Schema EMPRESAS**
   - Adicionar `COL_EMP_DT_ULT_REATIV = 21` em `src/vba/Const_Colunas.bas`, coluna U.
   - Atualizar cabeçalho de `EMPRESAS` e `EMPRESAS_INATIVAS` em `Mod_Limpeza_Base.MLB_CabecalhoEmpresas` (`src/vba/Mod_Limpeza_Base.bas:213-219`).
   - Atualizar todos os fixtures/cadastros que criam linha de empresa: `Repo_Empresa.Inserir`, cadastro direto no `Menu_Principal`, `TV2_CadastrarEmpresaCanonica`, reativação/inativação helper do Engine V2 e qualquer cópia por linha.

2. **UDT**
   - Adicionar `DT_ULT_REATIV` em `TEmpresa` dentro de `src/vba/Mod_Types.bas`.
   - Por ser TABU C4, tratar como microdelta dedicado com pre-flight de compile e import isolado. Não misturar com mudanças de UI no mesmo delta se o operador preferir risco menor.

3. **Repo_Empresa**
   - `LerEmpresa` deve ler a coluna nova e normalizar vazio/valor inválido para data zero.
   - Criar ou ajustar função de gravação para status de reativação com retorno explícito. Evitar que `Svc_Rodizio.Reativar` declare sucesso sem confirmar a persistência.

4. **Svc_Rodizio.Reativar**
   - Gravar `DT_ULT_REATIV` no momento da reativação automática ou manual de suspensão.
   - `AUDIT_LOG` de `EVT_REATIVACAO` deve incluir timestamp novo e status anterior.
   - Se a empresa já está ativa, decidir se a chamada deve ser idempotente sem atualizar `DT_ULT_REATIV` ou se deve rejeitar "reativação sem suspensão". Recomendo rejeitar ou no-op auditado, para não zerar janela indevidamente.

5. **Repo_Avaliacao.ContarStrikesParaPunicao**
   - Função nova, mantendo a existente intacta.
   - Se `DT_ULT_REATIV` vazio: contar como legado total.
   - Se preenchido: contar apenas avaliações concluídas da empresa cuja data de referência seja posterior ao timestamp de reativação.
   - Em erro, não retornar `0` silencioso para decisão de punição; registrar falha ou retornar resultado estruturado.

6. **Svc_Avaliacao**
   - Trocar apenas a decisão de suspensão para usar o contador de punição.
